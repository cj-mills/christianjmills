Template-Free Prompt Construction

  • Purpose: Provides flexibility in prompt design and avoids being locked into predefined chat templates.
  • Mechanics: Uses input, output, and label fields in the data.
    • Input: Represents user input to the model.
    • Output: Desired model response.
    • Label: Boolean indicating if the instance should contribute to loss calculation.
  • Advantages:
    • Simpler for users familiar with input-output data formats.
    • Suitable for multi-turn chat and platforms with existing input-output structures.
  • Disadvantages:
    • Less flexible when changing chat templates later.
    • Requires careful string handling at inference time to match training.
  • Recommendation: Use only if existing chat templates are insufficient.
  • Documentation: https://openaccess-ai-collective.github.io/axolotl/docs/input_output.html

Choosing Data Types for HuggingFace Datasets

  • Dataset Variety: Different data sets have varying formats for instructions, inputs, outputs, and system prompts.
    • Alpaca: Instruction, input, and output fields.
    • ShareGPT: Human/user and GPT/assistant turns, with variations in keys (e.g., role, from, value, content).
  • Axolotl Flexibility: Axolotl can handle various ShareGPT format variations automatically.
    • The sharegpt data set type infers the correct format from keys and imports data into a consistent message list.
  • Recommendation: Use the most appropriate data set type based on the data format and leverage Axolotl’s flexibility.

DPO and RLHF

  • DPO (Direct Preference Optimization): Simpler to implement, but limited to single-turn optimization.
  • RLHF (Reinforcement Learning from Human Feedback): More robust, allows for multi-turn optimization and intermediate rewards, but requires a reward model and more data.
  • Data Set Formats: DPO data sets typically include chosen and rejected responses for a given prompt.
    • argilla/dpo-mix-7k: Chosen, rejected fields.
    • Intel/orca_dpo_pairs: Question, chosen, rejected fields.
    • argilla/kto-mix-15k: Prompt, completion, label, rating.
  • User-Defined Data Sets: Offer flexibility for custom formats using YAML files to specify field mappings.
  • Axolotl’s Future Direction: Plans to incorporate more robust RL methods like PPO and improve dataset pipelines.

Chat Templates vs. Dataset Types

  • Chat Template: Defines the formatting of input and output sequences for the model (e.g., special tokens, separators). Stored in the tokenizer config.
  • Dataset Type: Specifies how Axolotl should process and structure the data from the dataset (e.g., ShareGPT, Alpaca). Affects how data is transformed into a common format (e.g., message list).
  • Interaction:
    • Setting a chat template in the YAML config automatically updates the tokenizer config.
    • Axolotl maps the dataset type to a specific output format (e.g., ShareGPT defaults to ChatML).
    • The chat_template parameter overrides the default output format for a dataset type.
  • Recommendation: Choose a chat template suitable for the desired interaction style and ensure consistency between training and inference.

Tokenization Consistency

  • Challenge: Ensuring consistent tokenization between training (Axolotl) and inference (e.g., VLLM).
  • Potential Issues:
    • Different libraries might handle string concatenation and special tokens differently.
    • Tokenization might be done on individual strings vs. concatenated strings.
  • Recommendations:
    • Use Chat Templates: Leverage specialized end-of-sequence tokens to minimize tokenization discrepancies.
    • Verify Tokenization: Carefully inspect and compare tokenization outputs between training and inference.
    • Introduce Tokenization Errors During Training: Potentially improves model robustness to minor variations.
  • Axolotl’s Approach:
    • chat_template parameter for handling tokenization more consistently.
    • Aims to align tokenizer configurations with common inference engines.

Multimodal Fine-tuning

  • Current State: Not currently supported in Axolotl.
  • Challenges:
    • Handling dataset pipelines for images and other modalities.
    • Integrating with multimodal architectures.
  • Future Plans:
    • High priority for future development.
    • Seeking contributions for dataset handling and pipeline implementation.

Conversational vs. QA Datasets

  • Key Takeaway: Choose the dataset type that aligns with the intended use case and production environment.
  • Conversational Datasets: Suitable for multi-turn chatbot scenarios.
  • QA Datasets: Sufficient for single-turn question-answering applications.
  • Recommendation:
    • If your application primarily involves single-turn interactions, QA datasets are sufficient.
    • Instruction tuning can provide valuable insights into how different dataset types influence model behavior.

Future of Axolotl

  • Need for Contributors:
    • Addressing the growing number of issues and feature requests.
    • Assisting with documentation and community support.
  • Development Priorities:
    • Building a user-friendly platform for streamlined fine-tuning and deployment.
    • Integrating more advanced RL methods like PPO.
    • Enhancing dataset pipelines and handling for various data formats.
  • Contribution Opportunities:
    • Join the Axolotl Discord server and contribute to discussions.
    • Explore the GitHub repository for open issues and feature requests.
    • Reach out to the maintainers for guidance on contributing.

Additional Notes

  • VRAM Estimation:
    • While some estimation methods exist, they become less accurate with techniques like FSDP and DeepSpeed.
    • A dedicated tool for accurate VRAM estimation would be beneficial.
  • Vibe Checks During Training:
    • Possible to pause training, infer against the checkpoint, and resume training.
    • Consider implementing a callback system for real-time predictions during evaluation steps.
  • Quick and Dirty Fine-tuning:
    • Use a small dataset like Tiny LLaMA.
    • Leverage Gradio for quick inference testing.
  • Function Calling Fine-tuning:
    • Requires specific ShareGPT versions with function calling role support.
